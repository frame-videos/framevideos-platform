# Security Documentation - Frame Videos

## Overview

This document outlines the security measures implemented in the Frame Videos platform to protect user data and prevent unauthorized access.

## Password Security

### Hashing Algorithm

**Implementation**: bcrypt with automatic salt generation

- **Algorithm**: bcrypt (industry-standard password hashing)
- **Salt Rounds**: 10 (balances security and performance)
- **Salt**: Unique per user, automatically generated by bcrypt
- **Previous**: SHA-256 (deprecated - not suitable for password storage)

### Why bcrypt?

- **Slow by design**: Makes brute-force attacks computationally expensive
- **Automatic salt**: Each password gets a unique random salt
- **Adaptive**: Can increase cost factor as hardware improves
- **Industry standard**: Widely tested and recommended by security experts

### Password Requirements

All passwords must meet the following criteria:

- ✅ Minimum 8 characters
- ✅ At least one uppercase letter (A-Z)
- ✅ At least one lowercase letter (a-z)
- ✅ At least one number (0-9)
- ✅ At least one special character (!@#$%^&*()_+-=[]{}|;:'",.<>?)

### Password Strength Levels

- **Weak**: Does not meet all requirements
- **Medium**: Meets all requirements (8+ chars)
- **Strong**: Meets all requirements + 12+ characters

## Rate Limiting

### Login Rate Limiting

Protects against brute-force attacks by limiting login attempts per IP address.

**Configuration**:
- **Max Attempts**: 5 failed attempts
- **Time Window**: 15 minutes
- **Block Duration**: 30 minutes

**Behavior**:
1. User exceeds 5 failed login attempts within 15 minutes
2. IP address is blocked for 30 minutes
3. All login attempts from that IP return HTTP 429 (Too Many Requests)
4. Block automatically expires after 30 minutes

**Response Example**:
```json
{
  "error": "Too many login attempts",
  "message": "Please try again in 25 minute(s)",
  "retryAfter": 1500000
}
```

## Account Lockout

### User Account Protection

Protects individual accounts from credential stuffing and brute-force attacks.

**Configuration**:
- **Max Failed Attempts**: 10 failed login attempts
- **Reset Window**: 30 minutes (attempts reset if no failures in this period)
- **Lockout Duration**: 1 hour

**Behavior**:
1. User account exceeds 10 failed login attempts
2. Account is locked for 1 hour
3. All login attempts return HTTP 423 (Locked)
4. Lockout automatically expires after 1 hour
5. Counter resets after successful login or 30 minutes of no attempts

**Response Example**:
```json
{
  "error": "Account temporarily locked",
  "message": "Too many failed login attempts. Account will be unlocked in 45 minute(s)",
  "retryAfter": 2700000
}
```

## Combined Protection

Rate limiting and account lockout work together:

1. **IP-based rate limiting**: First line of defense (5 attempts)
2. **Account lockout**: Secondary protection (10 attempts)
3. **Remaining attempts**: Response shows minimum of both limits

This dual approach protects against:
- Distributed brute-force attacks (multiple IPs)
- Credential stuffing (valid email, wrong password)
- Account enumeration attempts

## Authentication Flow

### Registration

1. Validate email format
2. Check password strength
3. Hash password with bcrypt
4. Create user account
5. Generate JWT token
6. Return user info + token

### Login

1. Validate input (email + password)
2. Check IP rate limit
3. Find user by email
4. Check account lockout
5. Verify password (bcrypt compare)
6. On failure: record attempts, return error with remaining attempts
7. On success: reset lockouts, generate JWT, return token

### Token Verification

- JWT tokens expire after 7 days
- Tokens include: user ID, email, tenant ID
- Signature verified using HMAC-SHA256

## Security Headers

The API implements the following security headers:

- **CORS**: Restricted to whitelisted domains
- **Content-Type**: JSON responses only
- **Authorization**: Bearer token required for protected endpoints

## Best Practices

### For Users

1. Use strong, unique passwords
2. Don't share credentials
3. Report suspicious activity
4. Enable 2FA when available (future feature)

### For Developers

1. Never log passwords or tokens
2. Always use HTTPS in production
3. Rotate JWT secret regularly
4. Monitor failed login attempts
5. Keep dependencies updated

If you discover a security vulnerability:

1. **DO NOT** open a public issue
2. Email security@framevideos.com (if available)
3. Include detailed description and reproduction steps
4. Allow reasonable time for response before disclosure

## Testing

### Password Strength Endpoint

Test password strength before registration:

```bash
POST /api/v1/auth/check-password
Content-Type: application/json

{
  "password": "TestPassword123!"
}
```

Response:
```json
{
  "valid": true,
  "strength": "medium",
  "errors": []
}
```

### Security Testing Checklist

- [ ] Password strength validation works
- [ ] Rate limiting blocks after max attempts
- [ ] Account lockout activates correctly
- [ ] Lockouts reset after successful login
- [ ] JWT tokens expire properly
- [ ] Invalid tokens are rejected
- [ ] CORS headers are correct

## Monitoring

Key metrics to monitor:

- Failed login attempts per hour
